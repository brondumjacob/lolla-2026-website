## Priority Issues

**[P0] Planner panel's `border-radius: 999px` breaks the entire dialog**
- **Why it matters**: Live-verified in browser — `.planner-panel` (schedule.html:337) applies a 420px-wide container's corner radius as a full pill. The header text renders as "LAN MY SCHEDULE" with the "P" clipped off, the close button (`#plannerClose`, line 490) is entirely invisible (confirmed via zoomed screenshot), and result rows get clipped too (a set time renders as "40" instead of "8:40"). This makes the site's best feature look broken and removes the one dedicated way to dismiss it.
- **Fix**: Change `border-radius: 999px` to `var(--radius-card)` (20px) on `.planner-panel` at line 337. Pill radius stays correctly reserved for `.planner-fab` and pill-shaped inputs.
- **Suggested command**: `/impeccable harden`

**[P1] Zero heading elements anywhere on the page**
- **Why it matters**: Grep confirms no `<h1>`–`<h4>` tags exist. "BUILD YOUR SCHEDULE," "HOW IT WORKS," and every day label are plain `<div>`s. A screen-reader user gets a flat, unstructured read with no way to jump to "pick your day" — a total navigation failure for a page whose entire job is fast day selection.
- **Fix**: Make `.sched-title` an `<h1>`, `.sched-how-title` an `<h2>`, and each day label an `<h3>` inside its card link.
- **Suggested command**: `/impeccable layout`

**[P1] Nested-opacity contrast stacking makes "How It Works" and planner text fail WCAG AA**
- **Why it matters**: `.sched-step` (line 160) sets `background: rgba(255,255,255,0.5)` over the dark `#2B1B14` section, then `.sched-step-desc` (line 180) layers `color: rgba(255,255,255,0.5)` on top of that already-translucent card — computed contrast ≈1.9:1 against a 4.5:1 requirement. `.sched-step-title` (pure white on the same composited card) computes to ≈3.3:1, also failing. Same underlying mistake recurs at `.sched-day-headliner` (line 124, ≈3.2:1), `.planner-hint` (line 394, ≈3.2:1), and `.planner-set-time`/`.planner-set-stage` (lines 438/453, ≈4.4:1, just under threshold). This is the same low-opacity-text bug found sitewide in the `index.html` audit, here appearing as white-on-dark instead of dark-on-light.
- **Fix**: Raise these to at least opacity 0.6 against dark backgrounds (0.5 white-on-`#2B1B14` computes to ≈5.1:1 and passes — use that as the floor). For `.sched-step`, stop compositing two translucent layers: give the card a solid computed background so text-opacity math only accounts for one layer.
- **Suggested command**: `/impeccable harden`

**[P2] The day-color system is a promise the builder pages don't keep**
- **Why it matters**: The four accent hex values (schedule.html:236/246/256/266 — `#8B5CF6`, `#E91E8C`, `#F97316`, `#00BCD4`) exist only inline on this page, not as tokens, and don't match `--pink`/`--cyan` (`#E85BB8`/`#2FA79C`) used in `styles.css` and inside all four `schedule-*.html` builders (confirmed via grep — every builder's `:root` uses the same generic pink/cyan regardless of day). The hub visually teaches "Friday = this pink, Sunday = this cyan," then the builder you land on abandons that color entirely. Friday's `#E91E8C` also doesn't match the site's canonical pink anywhere else — an unintentional off-brand color, and one of the CLI scan's 24 findings.
- **Fix**: Either tokenize these four colors as shared variables and thread them into each builder's header, or drop the false promise and use one neutral/brand accent if the builders genuinely don't differentiate by day.
- **Suggested command**: `/impeccable polish`

**[P2] No date-awareness for a page whose core value is time-based planning**
- **Why it matters**: Zero `Date` logic exists — no check for whether "today" falls before, during, or after July 30–Aug 2, 2026. A user opening this mid-festival gets the identical static page as someone browsing in March: no "TODAY" badge, no de-emphasis of days that already passed. Given the product already stores starred artists in `localStorage` (`favorites.js`), the hub also throws away an easy win: it could show "You've starred 2 artists for Friday" per card, and doesn't.
- **Fix**: At minimum, add a "TODAY" pill on the current day's card when the date matches; consider surfacing per-day My Lineup counts on each card.
- **Suggested command**: `/impeccable delight`

## Persona Red Flags

**Jordan (Confused First-Timer)**: Understands the 4-card day layout immediately — dates + headliners are a good crutch. Gets stuck because "HOW IT WORKS" sits *below* the day picker: a decisive Jordan can click "BUILD →" before ever seeing the tap/star/route explanation. If Jordan does scroll down first, the near-illegible grey-on-grey text (Priority Issue #3) means the instructions may not register anyway.

**Casey (Distracted Mobile User)**: The planner FAB is thumb-zone aware (`bottom: 28px→80px` under 600px). But hitting the border-radius bug (Issue #1) on a small screen is worse than on desktop — "tap outside to dismiss" requires more precise thumb placement with less spare screen real estate. Mobile viewport rendering below 600px could not be visually confirmed this run (the resize tool didn't change the captured viewport), so the single-column layout is verified from CSS only, not visually.

**Riley (Stress Tester)**: Gibberish input to the planner ("asdkfj") correctly returns "No artists found" with no crash. Hits the "zero headings" gap immediately on any screen reader — no way to jump to the day picker via landmark navigation. Visiting during the actual festival dates hits the "no date awareness" gap (Issue #5) head-on.

**Morgan (Returning Planner — project-specific)**: *Profile*: already has artists starred in My Lineup from earlier browsing; opens `schedule.html` on their phone at the festival, between sets, to figure out what's next. *Red flags*: no "TODAY" indicator on any day card — Thursday through Sunday look identical regardless of the actual date. The hub never cross-references the `localStorage` favorites it already has against each day, so a returning user with real skin in the game gets the same blank slate as a first-time visitor in March.

## Minor Observations

- `.sched-day-card:hover` has a hover transform (line 101) but no matching `:focus-visible` state — keyboard users get only the unstyled browser-default outline.
- `.planner-close` (lines 359–367) has no `:hover`/`:focus-visible` style at all, independent of the clipping bug.
- The planner uses `role="dialog"` (line 487) but no `aria-modal="true"` and no focus trap — Tab likely leaks to page content behind it.
- No Escape-to-close on the planner panel, inconsistent with the hamburger nav on the same page, which does handle Escape (`nav.js:26–31`).
- `#plannerResults` has no `aria-live` region — screen reader users get no announcement when results populate.
- The FAB visually overlaps the "3. BUILD YOUR ROUTE" step card at default desktop scroll position.
- The CLI scan's remaining ~21 findings (mostly off-ramp font-sizes: 0.7rem, 0.85rem, 0.9rem, 1.1rem, 1.3rem, 1.6rem, 2.5rem, etc.) are minor individually but form the same token-drift pattern flagged in the `index.html` audit — worth a systemic pass rather than one-off fixes.

## Questions to Consider

1. If the four day-accent colors don't carry through to the actual builder pages, is the color-coding earning its keep, or is it decoration that happens to also be a design-system violation? Should the hub's color promise drive a change to the builders instead of being quietly dropped?
